Xsnow 的"抗议软件"风波与 Debian

作者：Joe Brockmeier 发布日期：2026年6月29日

xsnow 这款应用会在 X11 桌面上生成动画飘雪效果（以及其他有趣的装饰），它看起来并不像是表达政治立场的渠道。然而，xsnow 的维护者似乎在程序中嵌入了一条政治抗议：当程序语言被设置为俄语（"ru"）时，会触发一个彩蛋。有用户投诉称，Debian 的 xsnow 软件包应当移除这项功能，但 Debian 似乎并没有任何规则明确禁止此类特性。

投诉

6 月 14 日，Alexander Ivanov 向 Debian 开发邮件列表发信投诉：如果用户的语言设置为俄语，会以"显著更高的概率"触发"一个伪装的视觉元素（被标注为 'EXTRATREE' 的乌克兰国旗）"。相关代码位于 xsnow 3.8.3（打包进 Debian 13 "trixie"）与 3.8.6（unstable 版本）的 src/scenery.c 文件第 326 行起：

      #ifdef USE_EXTRATREE
            if (global.Language && !strcmp(global.Language,"ru") && drand48() < 0.3)
	     tt = MAXTREETYPE;
            if (drand48() < 0.02)
	     tt = MAXTREETYPE;
      #endif

也就是说，当语言设为俄语时，xsnow 更可能显示乌克兰国旗（src/Pixmaps/extratree.xpm）。xsnow 允许用户从几种语言中任选其一；当程序设为英语时，我没有看到任何乌克兰国旗，但它们大约会有 2% 的概率弹出。我确实看到了预期的装饰，比如月亮、雪花、树木、北极熊、飞鸟，以及驾着雪橇的圣诞老人。将语言改为俄语后，xsnow 立刻在桌面上与其他装饰一起摆放了数面国旗。下图展示了它在 Debian 13 上的样子。

DFSG 违规之争

Ivanov 声称该行为违反了作为 Debian 社会契约一部分的 Debian 自由软件指南（DFSG）。具体而言，他认为这种"定向行为"违反了"不歧视个人或团体"（DFSG 第 5 条）和"不歧视任何工作领域"（DFSG 第 6 条）的原则。

Ivanov 并没有提交 bug，而是"向更广阔的 Debian 开发社区"报告了这个问题。他之所以这样做，是因为 Debian 软件包维护者 Willem Vermin 同时也是 xsnow 项目的维护者，而且正是他最初引入了这一改动。

Chris Hofstaedtler 回复说 Ivanov 搞错了：DFSG 要求的并非软件本身"不歧视"，而是软件的许可证不歧视。"所使用的许可证似乎并未违反 DFSG 的第 5 或第 6 条。下次请仔细阅读 DFSG。"xsnow 主要基于 GPLv3 许可，尽管部分文件采用了其他 Debian 认可的许可证。

Russ Allbery 同样认为 DFSG 与此无关；他还警告，引用社会契约和 DFSG 会"把讨论变成抠字眼的规则游戏，而无视真正的问题"。不过，即便 xsnow 符合 DFSG，他也表示这种国旗显示可能是 Debian 不希望在归档中保留的内容：

一般来说，我认为以欺骗方式运作的软件——包括基于用户名、区域设置或其他任何用户无法合理预期会以这种方式改变行为的本地设置或信息、暗中改变行为——大概不是我们想放进 Debian 的东西。这是一条非常 slippery（湿滑）的斜坡，而且很可能制造大量闹剧，却收益甚微。

Ivanov 表示他理解 xsnow 并未违反 DFSG 这一点。但他追问：难道不该把这种隐藏功能当作一个 bug，或者应当被打补丁去掉，"以恢复该应用对所有用户一视同仁的中立、预期功能"吗？Bill Allombert 建议他提交一个 bug 报告，因为 Debian 开发列表并非投诉的合适场所。

目前看不会改动 xsnow

到目前为止，Ivanov 尚未提交 bug，这场对话似乎也已告一段落。我已邮件联系 Vermin，询问他是否知晓这一投诉，以及如果有人提交 bug，他是否愿意回退乌克兰国旗的显示。截至目前，我尚未收到回复。

即便 Vermin 可能不想改变 xsnow 的行为，Debian 软件包维护者过去也曾因技术原因被迫做出改动——或者因冒犯性内容而被移出归档。2025 年 8 月，在一段漫长的讨论之后，两个"冒犯性"的 fortune 软件包在 trixie 发布前被移除。2025 年 10 月，Debian 技术委员会裁定，上游 systemd 的一项改动需要被回退，因为新行为破坏了若干依赖 world-writable /run/lock 目录的程序。在每一种情况下，维护者被推翻之前都经历了大量的讨论与审议。

这并非要评判 xsnow 的彩蛋应不应该被移除；只是想指出，即便维护者很可能持不同意见，向软件包提交 bug 也绝非徒劳。作为一个项目，Debian 给予开发者在管理各自软件包方面极大的自由度，但在项目认为确有必要的极少数情况下，打包者的决定也可以被推翻。然而，要说服项目这样做，需要请愿者自己付出努力。仅仅在邮件列表上抱怨，不太可能带来任何结果。

Git 2.55 有什么新功能

作者：Jonathan Corbet 发布日期：2026年6月26日

Git v2.55.0-rc2 测试版本于 6 月 23 日发布，这表明最终的 Git 2.55 版本可望在近期发布。虽然这次 Git 更新缺乏颠覆性的新特性，但它包含了一些常规 Git 用户会欣赏的改进，包括轻松编辑提交历史的命令、更多格式化选项、针对 Linux 的 fsmonitor 支持等等。

当前版本 v2.54 新增了一个实验性的 git history 命令。它在许多方面类似 git rebase，都能重写历史中的提交，但有一些重要差异，主要是为了把变基（rebase）变得不那么令人提心吊胆。它可以作用于裸仓库（bare repository），这对某些人可能有用。或许更广为有用的是 git history reword，它可以用一条命令轻松编辑过往提交的 changelog。

git history 有一处行为值得一提：与 git rebase 不同，它会变基所有从被修改提交派生出来的本地分支，而不仅仅是当前分支。大多数时候这正是人们想要的，但如果在一个临时分支上试验，它也可能带来意外。当然，我纯粹是通过研读源码理解了这种情况可能发生，绝没有在玩耍时不小心改动了重要分支。使用 --update-refs=head 选项可以把变基限制在当前分支。

无论如何，git history 仍被标记为实验性，但它也在不断增加新功能；2.55 新增的是 git history fixup。可以把它看作 git history reword 的一个变体，它修改的是提交的正文（body）而非 changelog。假设某次提交（称之为 abcdef01）修改了某个文件（称为 foo.c），却忘记更新一条注释，我们总是可以事后用 git rebase 补救。另一种替代做法是：直接编辑 foo.c 完成想要的改动，然后：

      git add foo.c
      git history fixup abcdef01

指定的提交会被这次对 foo.c 的改动所修订，所有从它派生的分支都会变基到新版本之上。除非提供了 --reedit-message 选项，受影响提交的 changelog 不会被修改。不过，如果变基过程中出现了合并冲突，该命令会直接中止。

其他新增还包括 git format-rev 命令，它读取一个或多个提交 ID，并用提供的 "pretty" 字符串格式化它们。以下是手册页中的例子：

      git last-modified -- <files> | git format-rev --stdin-mode=text --format=%an

这会打印出每个给定 files 最后被哪位开发者修改。

Git 的 "fsmonitor" 是一个守护进程，监视当前目录并记录其中的变更。各种 Git 子命令（如 git status）可以查询 fsmonitor 守护进程，而不必扫描目录寻找改动的文件。在文件系统访问缓慢、仓库很大的系统上，使用 fsmonitor 守护进程可以显著加速。这一特性此前一直不支持 Linux 系统——直到现在。从 2.55 开始，fsmonitor 守护进程也有了 Linux 实现。

git push 子命令现在可以把改动推送到一个 "remote group"（远程组），也就是一组互相独立的 remote。组在 Git 配置文件中定义；类似这样的命令：

      git config remotes.new-remote "remote1 remote2"

会创建一个名为 new-remote 的组。随后：

      git push new-remote

会同时推送到 remote1 和 remote2。

本版本其他值得注意的改动包括：

  • 通过配置系统设置的 Git hook，现在可以配置为并行运行，从而整体上加快执行速度。
  • diff 生成器现在能理解更多 Lisp 方言。
  • git url-parse 子命令可用于拆分 URL 的组成部分；这很有用，因为 Git 对 URL 格式有一些其他解析器不理解的特别扩展。
  • 用于 git rev-list 和 git log 的 --max-count-oldest 选项，可以请求指定范围内最"老"的 N 个提交。

除此之外，当然还有大量性能优化、各种细微调整以及一长串 bug 修复。截至撰写时，本版本总共包含 486 个非合并提交，且似乎已经稳定。因此 2.55 版本可望很快发布，Git 社区也将进入 2.56 的开发周期。

Rhombus：灵活的元编程

作者：Daroc Alden 发布日期：2026年6月30日

类 Lisp 语言在元编程和灵活性方面历来引领世界。虽然许多现代语言都采纳了宏（macro）的思想，但类 Lisp 语言如 Racket 仍在不断推陈出新，努力让宏尽可能容易融入日常程序。另一方面，Lisp 极简的、基于括号的语法却很难适应——以至于 Lisp 有时被戏称为 "Lots of Irritating Silly Parentheses"（一堆烦人的傻括号）。Rhombus 是一门新编程语言，旨在兼得二者的长处：将 Racket 的元编程能力，与简洁的类 Python 语法以及合理的标准库默认值结合起来。

项目

这门语言是更庞大的 Racket 项目的一部分，后者"植根于学术界"。Rhombus 的 43 位贡献者中，最活跃的两位是犹他大学的 Matthew Flatt 和香港中文大学的 Wing Hei Chan。Rhombus 在犹他大学和香港中文大学都被用作教学工具；Chan 是独立为 Rhombus 做贡献的，实际上他并不在香港中文大学计算机科学系工作。尽管如此，这门语言的目标不止于学术练习。其开发由 Racket 编程语言基金会支持，该基金会力求让 Racket 和 Rhombus 适用于真实的职业化使用。尽管这门语言直到 6 月 22 日才庆祝自己的 1.0 发布，但已经有一些非学术用途，比如一个部分用 Rhombus 编写的 Economancy 卡牌游戏工具。

Rhombus 的核心采用 MIT 或 Apache 2.0 许可证，但它构建于 Racket 运行时之上，而后者在一些动态库中包含 LGPL 3.0 代码。复用大量 Racket 基础设施，意味着 Rhombus 已经拥有数量庞大的可用库，以及一个优化编译器。Rhombus 程序可以被解释执行、编译为字节码，或编译为原生机器码。

Rhombus 区别于 Racket（除语法外）的一点，是它对默认数据结构的选择。类 Lisp 语言常把单向链表作为核心数据类型，这会伴随效率问题。Rhombus 使用的是由不可变树结构支撑的列表，使许多操作只需对数时间。标准库中还提供扁平的可变数组、基于树的映射和集合，以及基于哈希的映射与集合。

语法

Rhombus 的语法与 Python 相当相似：基于缩进，并用 ":" 表示缩进块的开始。一个细微差别是，Rhombus 在 if 表达式及类似语境中使用 "|" 符号分隔分支。最清晰的说明方式大概是用一个例子（阶乘函数的实现）：

      fun factorial(n):
        if n <= 1
        | 1
        | n * factorial(n - 1)

偏向命令式风格的开发者可能更喜欢下面使用 for 循环的版本。注意它使用了包含上下界的区间 1..=n，而非更常见的排除上界的 1..n。

      fun factorial(n):
        def mutable p = 1
        for (i in 1..=n):
          p := p*i
        p

默认情况下变量是不可变的，因此需要用 mutable 绑定操作符让循环能够改变 p 的值。不过，这个例子还能写得更短。Rhombus 的 for 循环支持一个在类 Lisp 语言之外很少见的功能：可选的 "reducer"（归约器），用于把每次循环的结果合并成循环本身的一个最终值。math.product 归约器会乘起来循环返回的所有值，从而让人写出：

      fun factorial(n):
        for math.product (i in 1..=n):
          i

这种能力意味着，Rhombus 中相当于其他语言"列表推导式"的东西，只是一个使用了 List 归约器来把结果收集进列表的 for 循环。for 循环还可以用 all 或 any 归约器产出映射、集合乃至布尔值。在典型编程语言中，人们可能预期这种内置语法只适用于标准库提供的类型，而不适用于用户自定义类型。但 Rhombus 的宏可以用来扩展语言的几乎任何部分，包括创建新的归约器。多数程序大概不需要这么做，但在它能简化程序时，有这一选项是件美事。

宏

Rhombus 的宏使用单引号来生成表示一段 Rhombus 代码的语法对象（syntax object），并用 "$" 表示占位符。不同于用列表表示代码的传统类 Lisp 语言，Rhombus 的语法对象是它们自己的数据类型。这让它们能携带额外的元数据，比如行号和位置，进而让 Rhombus 产生有用的错误信息，比如能正确考虑宏的栈回溯。语法对象还携带控制名字如何解析的元数据，让程序员能够控制宏何时、以及如何从环境中捕获名字。卫生宏（hygienic）和非卫生宏都可以编写，不过默认是卫生宏。

作为一个小程序宏的例子，以下是用内置 when 宏实现 Ruby 后缀 if 所需的全部代码：

      import:
        rhombus/meta open // 定义宏所必需

      expr.macro '$action if $test':
        'when $test
         | $action'

      println("Hi!") if (foo == bar)

在某种意义上，这类语法对象让人想起 Template Haskell 采用的方法。但与 Template Haskell 不同，Rhombus 的宏可以定义在任何普通函数能出现的地方，并且能响应它们被使用时的语法语境。上面的例子就是一个可以在任何期望表达式的场合使用的宏。归约器宏用 reducer.macro 定义，此外还有用于自定义 import、模式匹配等处理的类似构造。

人们或许会以为宏可以运作的语法语境列表是固定的。但在 Rhombus 中并非如此——它把操纵语法语境的函数暴露给了用户。只要程序使用 Rhombus 基本的基于缩进的语法，程序员就可以自由定义自己的语法和表达式种类，只要他们认为必要。

其他扩展形式

这种灵活性的另一个例子，是 Rhombus 处理类型检查的方式。在某种意义上，Rhombus 并不是一门静态类型语言。不过，如果加上类型注解，它看起来可以像：

      fun factorial(n :: Int) :: Int:
        def mutable p :: Int = 1
        for (i in 1..=n):
          p := p*i
        p

      // 调用 factorial("foo") 会产生错误信息：
      // factorial: argument does not satisfy annotation
      //  argument: "foo"
      //  annotation: Int

Rhombus 允许程序员把任意静态信息附着到表达式上，并定义这些静态信息应当如何被推断、扩展、检查，以及如何被用来生成机器码。默认情况下，语言把注解视为渐进类型系统（gradual type system）的一部分，尽可能推断类型，然后据此提供错误信息并生成更好的机器码。但程序员完全可以用自己的静态信息类别去附着到 Rhombus 程序上。例如，可以定义一套注解系统来追踪敏感数据在程序中的流动，或者追踪函数的渐近运行时间。宏可以访问这些静态信息，从而根据推断值以不同方式编译表达式。

Rhombus 倾向于把语言的所有实现细节都暴露在语言自身之内，这源于该项目对"面向语言编程"（language-oriented programming）的承诺。对许多任务来说，用户无需动用宏；语言提供的普通工具已足以应付日常使用。对于那些 Rhombus 本来就不太合适的任务，无论出于何种原因，这门语言鼓励创建迷你的嵌入式领域特定语言（EDSL），让表达问题的核心逻辑尽可能简单。

在许多语言中，一定量的样板代码是必要的，尤其是那些最好用标准库不支持的数据结构或关系来描述程序的时候。Rhombus 允许程序员把样板代码写一次、写在宏里，然后专注于用对他们最有意义的任何方式表达程序，即便那需要深入审视循环的语义或类型系统的行为。许多这样的微型语言扩展可以在同一程序中共存，而附着到语法对象上的额外元数据，使得创建通用的代码格式化器和高亮器、能处理自定义绑定宏的"跳转定义"支持，以及现代语言所期待的其他便利成为可能。

为支持其可扩展性，Rhombus 拥有覆盖语言各个部分的详尽文档，从解析一直讲到编译。Rhombus 继承了 Racket 社区对详尽文档的偏好——当你需要调查语言某些细节时这是福音，但当你只需要一个简单的概览时，它又略显冗长。

Rhombus 的细分定位

总体而言，Rhombus 最出色的特性无疑是它的可扩展性和文档。在编译速度、执行速度、包生态、错误信息等其他方面，它处于稳健的中游：好到不碍事，但还没好到能盖过专注那些领域的现有语言。例如，即便带上全套类型注解，Rhombus 代码也不太可能在速度上挑战 C 或 Rust——它通常性能介于 JavaScript 和 Ruby 之间，尽管具体数字因程序和基准设置而异。

Rhombus 最可能派上用场的地方，是那些有自研语言、或者用现有方法表达起来很别扭的问题领域。程序员历来具备编写新专用编程语言的能力，但编写传统优化编译器所需的工作量极其巨大。编写一组 Rhombus 宏则要容易得多，并且免费获得一个现成的编译器、现成的库、可用的语法高亮、错误信息以及编辑器集成，同时保留了在必要 deepest 层级定制语言运作的能力。

内核安全

用分配令牌和 bootpatch-SLR 加固内核

作者：Jonathan Corbet 发布日期：2026年6月25日

人们在消除内核中可被利用的 bug 并防止引入新 bug 方面做了大量工作。不过，即便这些工作取得最大成功，内核在短期内也不可能摆脱这些 bug。因此，人们也持续关注如何加固内核，让已有的 bug 更难被利用。即将发布的 7.2 内核版本将包含一项改动，改变动态分配结构在内存中的布局，使它们更难被覆盖；与此同时，一个在启动时随机化结构体内存布局的项目则有相当长的时间线。

用分配令牌进行内存分区

内核的 slab 分配器负责小对象的内存管理。分配来自固定大小（大多是 2 的幂）对象的桶，许多相同桶大小的不同类型对象可以从同一个 slab 分配。这种安排能高效利用内存，但如此混杂不同类型对象也可能让内核更容易受到某些类型的攻击。如果能诱使内核溢出某一类型的对象，就可能访问或破坏完全不相关类型的其他对象。此外，堆喷射（heap-spraying）攻击可能被用来用已知数据填满内核大部分内存空间，从而让其他类型的 bug（比如错误的指针解引用）更容易被利用。

早在 2023 年，就曾合入一组补丁试图加固内核以抵御此类攻击；该机制通过将 slab 分配器用于满足请求的内存进行分区来工作。与其用一组 slab 提供（比如）64 字节的内存块，分配器会使用 16 组。任何给定请求所用的具体 slab 是随机选择的，但以保证同一调用点发出的分配总是使用同一 slab 的方式。如此一来，用堆喷射之类技术填满内存的能力被大大降低。用缓冲区溢出覆盖目标对象的能力也降低了；大多数对象类型会位于遥远的 slab 中，而且任意给定 slab 中对象类型的具体组合会每次启动都不同。

这种随机化的分区让许多攻击变得更难，但它是一种概率性防御。多数时候它把目标对象与脆弱对象分开，但 16 次启动中仍有 1 次会把它们放进同一个 slab。在多系统云部署中，某些系统肯定会以对攻击者友好的方式布局脆弱对象。 naturally，安全向的内核开发者希望做得比这更好。

Marco Elver 尝试做得更好的一个方案是基于类型的 slab 分区，它已为 7.2 版本合入。顾名思义，它的工作方式是根据被分配对象的类型（而非分配调用点的地址）来分隔分配。

为此，该特性利用了 Clang 23 编译器提供的"分配令牌"（allocation tokens）功能。简而言之，该功能提供一个新的编译器内建函数 __builtin_infer_alloc_token()，它从传入参数的类型生成出一个整数令牌值。该令牌进而可被用来选择对象将从哪个分区分配。结果是，相同类型的对象总是从同一分区分配，即便有多个调用点在进行分配。一个涉及漏洞的结构体将总是被限制在单一分区内，并总是与大多数其他结构体类型分隔开。

作为一项额外的加固特性，令牌的生成会考虑相关类型是否是包含指针的结构体；表示含指针类型的令牌空间与所有其他类型的空间互不相交。因此，不含指针的类型与含指针的类型保持分离，使得用这些类型进行的任何溢出漏洞都更难被用于覆盖指针。

这种方法的潜在缺点是，从类型到分区的映射现在是确定性的、因而可被攻击者预测。旧有的随机化分区仍作为一个配置选项可供偏好者使用。不过，发行版维护者将不得不做出选择：他们想为用户启用哪种加固变体（如果有的话）。

运行时结构体布局随机化

攻击者可以从他们掌握的、关于特定系统内存布局的任何信息中获益。这包括结构体内字段的布局；若没有这些信息，运行中的系统中攻击目标的定位会更难。结构体在源码中布局，而 C 标准对结构体声明应如何映射到内存中的结果表示相当明确。攻击者会利用这些信息，但如果内存中的布局以不可预测的方式偏离了源码，他们就可能被迷惑。

这并非新想法；内核早已支持 "randstruct" GCC 插件，它能随机化结构体内部字段的顺序，至今已有多年。不过，randstruct 是一个编译期操作；它在内核构建时重排结构体字段。对运行自研内核的站点来说，编译期随机化或许足够。但对发行版发布的内核而言，它帮助不大；攻击者可以查看内核镜像，从而知道运行该特定内核的所有系统中结构体将如何布局。

York Jasper Niebuhr 最近带着一个名为 Bootpatch-SLR 的早期补丁集登场，旨在解决这一缺陷。它在启动期执行结构体随机化，意味着每个内核都会有不同布局，且每次系统重启布局都会改变。这应该会让任何试图瞄准特定结构体字段的人日子更难过。

Bootpatch-SLR 的工作原理是用一组特殊宏在源码中标注要随机化的结构体；下面这个标注 struct task_struct 的补丁展示了它如何运作。简而言之，一组要随机化的字段用如下标记标注：

      struct foo {
          spslr_struct_fields_start
  	/* 要随机化的字段放在这里 */
  	spslr_struct_fields_end
      };

这些标记会变成特殊的 ELF 段，被包含进最终的内核二进制中；它们不仅标记了要重排的字段，还标记了对那些字段的所有引用。当内核启动时，它读取这些段，重排指定的结构体字段，并通过一堆神奇的"代码补丁"更新所有引用以匹配。

自然，有些地方字段不能如此轻易重排。内核中许多地方把一个结构体作为容器结构体的第一个元素嵌入，然后假设指向容器的指针可以被转换为指向嵌入结构体的指针。重定位这个嵌入结构体很可能会产生比预期更"随机"的结果。其他结构体的布局可能由协议规范或硬件决定。完全不应被改变的结构体可以简单地不做标注；不过，另一些结构体可能能容忍部分字段重排，而其他字段保持不变。针对那种情况，有一个特殊标记 __spslr_field_fixed，可用来标记需要在结构体内保持声明位置的字段。

这个想法很有吸引力；一旦完全实现，它能显著增强现有内核的抗攻击能力。不过，那目前还是一个相当遥远的目标。除此之外，Bootpatch-SLR 是作为 GCC 插件实现的，而长期计划是从内核构建系统中移除对这些插件的支持。现在尝试新增一个插件，成功的可能性不大。还有许多其他细节需要厘清；其中若干在 Bootpatch-SLR 详尽在线文档的"已知问题"一节中有描述。

时间会告诉我们 Bootpatch-SLR 能否达到能被合入主线内核的境界；历史上充满了从未到达那一境界的绝妙想法和有趣原型。但如果它真的到了那里，安全收益可能会相当可观。

更早触发写回（LSFMM+BPF 峰会）

作者：Jake Edge 发布日期：2026年6月26日

写回（writeback）是确保页缓存中脏页或 folio 被刷入磁盘、从而让这些文件的改动得以持久化的过程。在 2026 年 Linux 存储、文件系统、内存管理与 BPF 峰会上的一场文件系统相关会议中，Jeff Layton 想讨论是否应当比今天更早发起写回操作。共识似乎是应该更早做，但实现路径却不那么清晰。

背景

Layton 开篇道，过去几个月里，一些新的 ioctl() 命令被加入 NFS 服务器（NFSD）的 debugfs 接口，提供了执行 "dontcache" I/O（与 RWF_DONTCACHE 相关，它原本叫 RWF_UNCACHED）和直接 I/O（O_DIRECT）的能力。这些控制被加在 debugfs 中，是为了给开发者一个机会在各种事情最终确定之前先试验。Layton 说，大部分工作是 Mike Snitzer 完成的。最终加入的 debugfs 接口与最初提案略有不同。

人们用新的 I/O 模式做了各种实验，试图量化哪些工作负载会从中受益。有一件事很快就清楚了：dontcache "对写入来说相当糟糕，至少在 NFSD 下是这样"。原因有不少，但主要是 NFS 客户端会并行发送大量写入，最终由服务器中的多个线程写出；那并不适合 NFSD 的那种 dontcache 实现。

他有一些在途的补丁修复了 RWF_DONTCACHE 用户的性能问题；他感谢了 Jan Kara、Christoph Hellwig 和 Ritesh Harjani "帮我把它打磨成型"。峰会之后，这些补丁已被合入 7.2。dontcache 的一个特征是它立即把写入提交给设备，而这正是 NFSD 遇到的问题之一；所有线程都被唤醒并提交 I/O，导致它们排队等待才能提交。修复办法是把数据放进页缓存，直到写回完成，并把写回操作移交给 flusher 线程；Layton 说，这种模式现在在许多情况下比直接 I/O 表现更好。

这项工作让他开始思考：一般性地更早做写回。他的修复之后累积的脏页数量低得多，这引出一个问题："在常规缓冲 I/O 中，我们是不是等了太久才启动写入？"写回在内存管理子系统判定脏页数量过高时被发起，也会周期性地发起。但"现代内存容量巨大"，所以所用的默认脏页上限已经过时了；他认为这些上限已有 20 年或更多未曾更新。这其实是个独立的问题，而且管理员"可以通过调优绕开"，但考虑改变默认值或许有意义。

他想知道，依赖内存管理来决定何时做写回是否真的是正确思路。也许文件系统层应当更主动一些；可以基于后备存储和发往它的脏页数量做某种计算，来决定何时发起写回。Linux 的哲学是空闲内存就是浪费的内存，这也是页缓存存在的原因，但闲置的磁盘带宽同样是一种浪费，这也是事实。

他建议设定一个比内存管理通常所用低得多的每超级块（per-superblock）脏页上限。更早启动写回"有助于让我们在脏页方面不陷入泥潭"。在会议之外，Kara 指出一个潜在问题：有些工作负载会创建大量临时文件然后删掉它们。这些文件通常在被写入之前就被删除了，所以更早启动写回可能导致它们被不必要地写出。Layton 认为，如今很多这类工作是在 tmpfs 上完成的，所以问题或许没那么严重。Kara 曾建议跳过那些没有目录项（dentry）的 inode 的写回，这可能也是一种规避大部分问题的方法。

讨论

Layton 说，他只是把这些想法抛出来供讨论；他并没有关于该往哪走的固定看法。"有什么想法吗？"

Ted Ts'o 说他有些想法，但可能有些自相矛盾。他好奇内核通过不频繁唤醒磁盘来节省电池电量有多重要；他指出如今还在用这类磁盘的笔记本已经很少了，所以笔记本模式或许已成为过去式。（结果笔记本模式已在 7.0 内核中被移除。）

还有某些种类的文件"一定程度的延迟是有益的"，因为这能让应用程序完成文件的写入，而这也对文件系统进行延迟分配时有帮助。所以也许那些仍然打开、仍在被写入的文件应该被推迟写回一段时间。然而，这类事情有很多策略问题需要解决。

Ts'o 说，另一个想法可能是尝试以后备设备带宽的 50% 进行写入，如果这能被计算出来的话，以确保留有余量，让 fsync() 延迟相当低。可以想象一个自动调优过程，为 fsync() 繁重的工作负载预留更多余量。正如他所指出的，他的一些想法"彼此之间是有张力的"，所以他更希望所用的启发式是自动的。依赖人类去改变各种参数不是好策略：它"永远不会发生"。Layton 指出，为写入预留太多后备设备带宽可能给读取留下太少，所以这又是一块拼图。

Boris Burkov 说他在一个巨大内存的系统上遇到过一个问题，累积了大量脏页；"它把写回堆积到极其庞大，最终的写回糟透了"。3 月 2 日，他尝试了一个把固定脏页上限调低的改动，但第二天就不得不全部回退，"因为它搞坏了一切"，他引来一阵笑声。他并不是说改默认值是个坏主意，因为默认值也很糟，"但你会搞坏某样东西"。

Kara 问用更低上限时具体遇到了什么问题。Burkov 说那是一个下载并安装软件包的系统；上限改变后它的性能退化了。Kara 推测这可能就是他早先向 Layton 提出的临时文件问题；也许安装过程用了大量临时文件，在更高上限下它们其实不会被写出。

Chuck Lever 提出了与 NFSD 相关的"一个相反的问题"。有些情况下 NFS 客户端发送了如此多的写入，以至于 NFSD 根本跟不上，"而且似乎服务器对客户端没有有效的反压（pushback）"。有一些方法可以通过 NFSv4 或调整 TCP 窗口来放慢速度，但他认为需要一种让虚拟文件系统（VFS）层能放慢写者的方法。Neil Brown 曾告诉 Lever 已经存在做这件事的机制，但也许它没能正确工作。"这个房间里坐满了专家，谁都可以告诉我我错了，来吧"，Lever 咧嘴一笑说道。

Kara 说他"某种程度上是对的"，因为反馈机制是面向本地文件系统的，在那里可用于写入的资源受限于与写回机制共享系统的生产者。但对 NFS 来说，客户端能发送的写入数量只受内存限制。当写入数据过多时，有机制可以节流写者，但对 NFS 来说它显然是坏的。"来自服务器的反压没有正确地传播"到客户端。Kara、Layton 和 Lever 之间一场快速推进的对话似乎得出结论：有办法提供这种反压。

Lever 问有什么办法能观察到反压以及随之而来的减速。Kara 说没有，但可以构建；他在哪些信息可以被导出以帮助观察方面提了一些建议。

Ts'o 说，基于文件名和其他属性预测文件生命周期的研究有几十年的积累，这可能有助于判断哪些文件应该被推迟写回。那将是一个启发式，要么需要系统管理员的决策，"这永远不会发生"，要么需要某种自动判定，或许基于部分研究。他说，对这一过程拥有某种可观测性也很重要，"因为调试这个将要求我们理解究竟是哪些神秘启发式触发了——无论对错"。

Christian Brauner 担心内核里任何"适合所有现存工作负载"的自动启发式配置"注定会失败"。他指向各种控制组（cgroup）参数，它们让内核"把这个问题变成用户空间的问题"，而不必想出一个覆盖所有人需求的东西。他认为控制组和 systemd 使用的机制，为写回上限之类各种问题应如何被处理提供了一个良好范本。另一方面，"我们有 fadvise()，尽管没人用它"，Ts'o 引来一阵会意的笑声。

Lever 向 Layton 追问了最初 NFSD 的 dontcache 方案为何表现不佳的更多细节。Layton 说，基本问题是：立即从接收客户端请求的线程执行写入，会阻塞该线程直到写入完成。通过把那部分工作移交给 flusher 线程，服务器线程就能在写回于 flusher 中发生的同时继续处理请求。

Lever 还想知道一旦写回完成，页缓存项是如何被丢弃的，因为那正是 RWF_DONTCACHE 的用意所在。Layton 说，有一个 folio 标志 PG_dropbehind，跟踪写回完成后是否把 folio 留在缓存中。RWF_DONTCACHE 写入与缓冲写入之间唯一真正的区别，就是这个位的设置，以及为 RWF_DONTCACHE 尽可能早地发起写回。

又讨论了一两分钟其他快速话题后，会议时间耗尽，就此结束。

BPF 程序对本地存储的高效访问（LSFMM+BPF 峰会）

作者：Daroc Alden 发布日期：2026年7月1日

当一个 BPF 程序被用于在网络子系统中过滤或重定向数据包时，该程序通常希望随着数据包在内核中移动而把数据与它关联起来。内核的本地 BPF 存储（local BPF storage）API 能把额外数据与某些内核对象关联起来，从而提供了实现方式。（也请参阅以 STORAGE 结尾的 BPF map 类型。）Amery Hung 和 Jakub Sitnicki 在 2026 年 Linux 存储、文件系统、内存管理与 BPF 峰会上主持了两场会议，讨论如何让对本地存储数据的访问更高效。Hung 谈的是与锁相关的一般性能问题，而 Sitnicki 特别考察了本地存储在网络子系统中的应用。

Hung 过去八个月一直在做一项利用本地存储实现内核与用户空间更快速通信的特性。这场演讲不是关于那个特性，因为在做它的过程中，Hung 注意到 BPF 自测有时会因为保护本地存储免遭死锁的锁发生竞争而失败。这种竞争可能导致从本地存储读取或写入失败，这显然是个问题源头。

本地存储由两把锁保护：一把用于保护并发访问的普通锁，以及一把每 CPU 锁——它的存在是为了防止死锁。需要后者是因为，一个 BPF 程序可能被挂载到在访问本地存储时会被触发的 tracepoint 上。如果没有每 CPU 锁来确保同一 CPU 不会试图获取主锁两次，那么挂载到其中一个 tracepoint 的 BPF 程序就可能试图访问本地存储自身，从而让内核死锁。为了防止这一点，BPF 用于访问本地存储的 kfunc 被做成了可能失败（fallible）的，而每 CPU 锁确保当两个不同的 BPF 程序试图同时访问本地存储时它们会返回一个错误。

不过，这比必要的更保守，因为它意味着两个互不相关的 BPF 程序即便完全不可能死锁，也会互相阻塞对方的访问。为解决这个问题，Hung 想用一种弹性排队自旋锁（rqspinlock）替换主锁——这是 Kumar Kartikeya Dwivedi 和 Alexei Starovoitov 开发的一种锁，用于在运行时报告死锁。既然主锁会防止死锁，每 CPU 锁就可以被消除了。只要小心，这种办法可以做到只在确实存在死锁风险时才让 BPF 操作失败。

但事情并非只是改一下所用锁的类型那么简单，Hung 说。这对大多数代码路径足够，但有些"不可能失败"（infallible）的操作需要拿同一把锁，而 rqspinlock 处理不了那种情况。例如，释放一个带有本地存储的内核数据结构，不应该仅仅因为锁竞争就失败。要让事情优雅地失败很难。

Hung 尝试用 kmalloc_nolock() 替换一些相关的分配与释放，但效果也不好。问题在于本地存储能用两种分配器：BPF 分配器和 slab 分配器。单独用其中之一在性能上不可接受；BPF 开发者们经历了很多痛苦才把二者解耦，使得任何一种都能被使用。

最终，Hung 放弃了优雅地解决该问题，并为本地存储引入了一个最坏情况下泄漏 176 字节的"不可能失败"释放操作。有了基本可行的方案后，下一步是对改动做基准测试。一个创建本地存储分配、然后在循环中释放它们的微基准，在移除每 CPU 锁后于 x86_64 上看到了 1.5% 的吞吐提升。Arm 系统则没有可测量的性能差异。不过，微基准并不能说明全部，因为真实世界性能对系统其余部分的状态很敏感。

这一改动还解锁了一些其他潜在的性能改进，包括把某些 slab 分配器的用法切换为能够使用 sheaves，这给另一个微基准带来了 5.1% 的加速。一位听众问 Hung 是否知道为什么两种分配器之间会有如此大的性能差异。Hung 不太确定——x86 上的缓存使用增加了，但那并没有导致性能回退。

接下来的步骤有些不明朗；Hung 说，围绕本地存储对象生命周期管理还有一些额外的复杂性。某些种类的内核对象并不总是完全初始化它们的本地存储，这可能导致内核崩溃。Starovoitov 此前曾建议用一个更通用的 map、以内核对象指针作为键，来作为本地存储的替代，那是个或许有趣的解法。

Hung 开发了另一个基准，把这种方法（使用 50,000 个桶大小的哈希表）与现有本地存储实现比较，并看到了加速，但在场的 BPF 开发者们表示怀疑。一位听众评论说，据他的经验，更小的哈希表更快、更大的哈希表更慢；恰当的基准应当测试多种大小。Hung 指出 rhashtable 使用的是动态桶数量，并好奇那会如何影响结果。

Sitnicki 说，在他工作的系统上，本地存储配合大约一百万个并发连接使用，这个数字让一些人感到惊讶。Hung 同意基准大概需要扩展。另一部分原因是，不同种类的内核对象看重不同的指标；用于 sk_buff 结构体的本地存储应当针对创建速度优化，等等，Hung 说。

将来，套接字本地存储还能通过彻底去掉动态分配的需求来进一步优化。验证器或许能做"helper fixups"让它更快，Hung 没有对此详述。到这时，会议已演变成了关于究竟该如何为不同用途对本地存储性能做基准测试的讨论。

数据包的元数据存贮

第二天，Sitnicki 主持了一场后续会议，讨论在 sk_buff 结构体（SKB）中存储元数据——这也是 2025 年讨论过的主题。"它不完全是本地存储，但这个话题应当让人感到熟悉。"他说，BPF 程序常常需要在数据包上存储标签、时间戳或其他少量数据。他的第一、二次尝试是向 BPF 程序暴露现有的快速数据路径（XDP）和 SKB 元数据；那不可行，因为这些元数据只能从 XDP 和流量控制（TC）语境访问，而不能从网络栈的上层访问。

要让这些元数据持久存在，需要网络子系统承诺一个固定的布局，而网络维护者们并不希望那样做。John Fastabend 质疑网络界人士为何会如此坚持。他问，问题是不是他们希望元数据在不同硬件间保持一致。Sitnicki 不认为是这样——人们可以从 XDP 分配元数据空间，稍后在 TC 消费它，而与硬件无关。

Sitnicki 考虑过几个替代方案；例如，可以用一个 SKB 扩展来持有指向本地存储的指针。但那会导致每个数据包三次分配，实在不可接受。另一个替代方案是专为 BPF 元数据设立一个 SKB 扩展。该扩展甚至可以与 SKB 位于同一次分配中。这意味着每个数据包一次分配，以及更少的间接层来访问数据。另一方面，它需要网络维护者的认同，会导致更大的 SKB，而且即便用不上也要为那部分内存买单。Sitnicki 打算在 7 月的 Netdev 会议上把这个方案交给网络维护者们斟酌。

Starovoitov 建议用一个启动标志来设定元数据的大小；这样不开启该标志的人看不到任何内存开销，而开启了的人大概知道自己在于什么。Andrii Nakryiko 问 Sitnicki 为何一开始想把元数据放在 SKB 旁边，尤其是访问它在实践中仍会导致一次缓存未命中（cache miss）。他建议另加一个用于元数据的独立哈希表来使用。Sitnicki 同意那是一个第二可能的替代方案。

Daniel Borkmann 让 Sitnicki 详述一下：让现有 SKB 元数据在网络栈更高层可访问，与在 SKB 扩展中加入 BPF 专用元数据，这两者区别何在。两种方案实际上都会把元数据放在 SKB 旁边。Sitnicki 同意确实如此，且两种他都可以接受。区别在于，使用 SKB 扩展的方案会有一层指针间接，即便只是一次分配。那更灵活，因为对于不需要元数据的数据包，可以把它省掉，从而节省内存。

不过，使用 SKB 扩展确实引入了一些新挑战。首先，它影响了 SKB 应当如何被克隆，因为扩展也需要被复制——而那不是当前 SKB 扩展的工作方式。Sitnicki 说，现有的 SKB 扩展只在激活一个新扩展时才被复制。此外，有些 SKB 扩展即便不需要也很难编译掉；例如，IPSec 支持总是需要为 IPSec 硬件卸载所用的 SKB 扩展留出空间。

Starovoitov 不确定为什么所有 SKB 扩展都需要与 SKB 在同一个分配中。Sitnicki 解释说，SKB 扩展就是那样实现的。任何试图只为 BPF 添加独立机制的方案都会被网络维护者拒绝，因为他们一直在努力让 SKB 扩展"不那么糟糕"很长时间了。他们希望这一领域的任何改进都能与现有代码协同工作。

不过还有另一个问题要考虑。即便预留了一些内存给 BPF 元数据，如何在所有都想往里存东西的 BPF 程序之间分配这块内存？一个选项是：把每个打算运行的 BPF 程序配置为拥有自己的范围。另一个选项是在元数据中动态分配空间，让 BPF 程序声明自己需要多少元数据，然后内核透明地提供访问。第二个用户体验更好，但实现更复杂。关于两种方案利弊有一些未能达成共识的讨论。

Sitnicki 说，另一个替代方案是把指向 SKB 的指针作为键，用在现有的 BPF 哈希表中。这听起来是个好方案，但为哈希表挑选正确大小是个棘手问题。也许 rhashtables 这种可调整大小的才是正确的答案，但那需要在真实负载下测试。使用哈希表还把"SKB 释放时清理元数据"的责任转移到了 BPF 程序身上。挂到一个 tracepoint 来检测 SKB 何时被释放确实有一些开销，需要作为该方案成本的一部分加以考量。

Starovoitov 质疑，为什么作为元数据最终消费者的那个 BPF 程序不能做清理，从而避免 tracepoint 的开销。Sitnicki 解释说，SKB 可能在网络栈中许多不同的点消失。如果一个带元数据的数据包被丢弃了，那元数据需要以某种方式被清理。Starovoitov 建议在 SKB 中放一个单独的位来指示它是否带有元数据，然后有条件地调用 tracepoint，这样就只对那些带元数据、且在能被清理前就被丢弃的数据包产生性能开销。

Nakryiko 和 Starovoitov 鼓励 Sitnicki 在决定设计之前，先把他的用例中需要元数据的确切数据包比例、元数据大小、以及数据包总体数量确定下来。所有这些都会影响最佳方案的样子，而且具体化比试图设计一个可能根本招不来其他用户的通用机制要好。到这时，讨论已沦为对各种设计影响的猜测，会议时间耗尽。

7.2 合并窗口的其余部分

作者：Jonathan Corbet 发布日期：2026年6月29日

Linus Torvalds 于 6 月 28 日发布 7.2-rc1 并关闭了 7.2 合并窗口；到那时，已有 13,412 个非合并提交进入了主线。这使得这是自 2024 年 6.7 开发周期（15,418 个提交，其中包括整个 bcachefs 开发历史的 2,800 个）以来最繁忙的合并窗口。其中略少于一半的提交是在 LWN 撰写合并窗口前半段总结之后到达的。像往常一样，合并窗口后半段的提交更聚焦于修复，但仍有大量新特性和重大改动被合入。

一些最有趣的合入改动包括：

核心内核

  • 持续改进内核交换（swap）子系统的工作仍在继续，合入了一系列统一处理匿名内存与共享内存 folio 的代码，并降低了交换子系统自身的内存占用。"静态元数据开销现在接近于零，工作负载性能略有提升"。详见我们的 LSFMM+BPF 2026 报道。
  • khugepaged 内核线程现在可以自动创建多尺寸透明大页（mTHP）。详见本报道。
  • 移除了一项为只读文件在页缓存中创建透明大页的独立选项——这是针对那些原生不支持大 folio 的文件系统的权宜之计。这一改动在那些文件系统获得大 folio 支持之前，可能会拖慢某些文件系统的性能。

文件系统与块 I/O

  • 在至少有 16GB 内存的系统上，NFS 传输的默认块大小已增大到 4MB。
  • SMB 文件系统服务器已支持以压缩格式存储的文件；它现在也能对通过网络发送（或接收）的数据实施压缩。
  • 新近恢复活力的 NTFS 文件系统已加固以抵御多种类型的磁盘元数据损坏，现在还能处理 Windows 原生符号链接。
  • 为 EROFS 文件系统提供支持的 fscache 后端（于 5.19 合入）已被弃用两年，现在被移除。
  • Ceph 文件系统获得了手动重置客户端的能力。这一特性严格来说没有文档，但补丁系列中包含一封信息丰富的封面信，而在系列被应用时那封信没有被保留下来。

硬件支持

  • 时钟：Canaan Kendryte K230 时钟、Qualcomm IPQ9650 全局时钟控制器、Qualcomm Hawi TCSR 时钟控制器、Qualcomm X1P42100 视频时钟控制器、Qualcomm X1P42100 相机时钟控制器。
  • GPIO 与引脚控制：Qualcomm IPQ9650 引脚控制器、Qualcomm Nord 引脚控制器、Qualcomm SM6350 LPASS LPI 引脚控制器、Qualcomm Shikra 引脚控制器、Aspeed G7 SoC 引脚控制器、NVIDIA Tegra238 与 Tegra264 引脚控制器、UltraRISC DP1000 SoC 引脚控制器。
  • 工业 I/O：Analog Devices AD5706R 数模转换器、MEMSIC MMC5983 三轴磁传感器、Broadcom APDS9999 ALS/RGB/接近传感器、Analog Devices AD4691 模数转换器、Vishay VEML3328 RGBCIR 光传感器。
  • 输入：Rakk 游戏外设、OneXPlayer 手持控制器、Wacom W9000 系列触控笔屏。
  • 媒体：AMD ISP4 图像信号处理器、AVMatrix HWS PCIe 图像采集设备、Intel CVS CSI-2 桥接器。
  • 杂项：Renesas R-Car 多功能接口、NVIDIA Tegra114 外部内存控制器、Verisilicon I/O 内存管理单元、TI LP5860 LED 控制器、Samsung S2M 系列 RGB 与闪光/手电 LED、Maxim MAX25014 控制器、Microsoft Surface RT 电池、Samsung S2M 系列 PMIC 电池充电器、AMD Promontory 21 xHCI 温度传感器、Qualcomm SHIKRA/HAWI/Nord 互联提供者、Qualcomm Hamoa/Glymur 参考设备嵌入式控制器、Dell DW5826e PLDR 复位控制器、UltraRISC PCIe 主机控制器。
  • 网络：Virtio CAN 接口。
  • PHY：Axiado eMMC PHY、Mobileye EyeQ5 以太网 PHY、EcoNet PCIe PHY、TI DS125DF111 双通道重定时器、Freescale Layerscape Lynx 10G SerDes PHY、NXP TJA1145 CAN 收发器 PHY。

杂项

  • perf 工具照例收到一长串改进；详见这个合并提交。

安全相关

  • 完整性度量架构（IMA）现在能够把度量数据暂存到内核之外，从而释放（可能是大量的）内核内存。详见 Documentation/security/IMA-export-delete.rst。
  • Landlock 安全模块现在有能力控制 UDP 套接字的使用；这个文档提交展示了改动内容。现在还可以抑制某些 Landlock 拒绝（denial）的日志，这个提交包含更多信息。
  • 经过六年努力，内核中的 strncpy() 实现已被移除；根据这个合并提交，这项工作耗费了来自 70 位贡献者的 362 个提交。这个提交列出了内核现在使用的替代方案。

未来走向尚不明朗；Blaise Boscaccy 和该领域的其他开发者显然认为 Hornet 提供了内核内解决方案所不具备的安全保证。Danial Borkmann 发布了一个替代实现，他希望那能构成一个可接受的折中。不过那项工作处于早期状态，近期不太可能进入上游。

虚拟化与容器

  • KVM 子系统已支持 Intel 的"基于模式的执行控制"（MBEC）和 AMD 的"客户机模式执行陷阱"特性，二者都对客户机内的执行权限提供更好控制。除此之外，这一特性大幅减少了客户机为权限相关操作而不得不陷回宿主机的次数。详见这个（合入前被大幅重写的）MBEC 补丁系列和这个合并提交。
  • 这次 Arm 架构没有新的 KVM 特性；来自 KVM 拉取请求："在 KVM/arm64 方面，这是个有点奇怪的合并窗口。拉取请求里绝对没有任何新特性。它纯粹是修复，因为当这么多 AI 催生的修复涌上列表时，审查新东西变得太难了。"

内核内部改动

  • 内核构建系统现在能够利用已被加入内核源文件的 SPDX 信息，创建软件物料清单（SBOM）。只有当前配置下实际被编译的文件会被考虑，因此 SBOM 是针对所构建内核特定的。相关命令是 make sbom；详见 Documentation/tools/sbom/sbom.rst。

7.2 合并窗口移除了 195 个导出符号、新增了 380 个其他符号；还新增了 32 个 kfunc。完整列表见此页。

最后，7.2 合并窗口包含了来自 2,138 位贡献者的贡献，这几乎肯定是一个合并窗口纪录。只有最近三个开发周期才在整个周期中见过这么多贡献者。在 7.2 合并窗口的贡献者中，404 位是首次贡献者。有 706 个提交（约占总量的 5%）包含 Assisted-by 标签。看来，LLM 催生的内核补丁洪流没有任何放缓的迹象。

现在所有这些改动都需要为 7.2 发布做稳定工作，而那最可能发生在 8 月 16 日。

Secure Boot 证书过期已经来临

作者：Brian "bex" Exelbierd（特约撰稿人） 发布日期：2026年7月1日

开启了 Secure Boot 的 Linux 用户依赖微软签发的证书来验证用于启动系统的软件是否受用户信任。其中一张证书最近已过期，但那不会导致已经能启动的系统停止启动。不过，在某些情况下过期可能引发问题，而且依赖现有已签名二进制的窗口比看起来要短。用户和管理员需要密切关注这些变化。过去一年里，我在微软工作的一部分就是处理这个问题。LWN 曾在 2025 年 7 月写过证书过期，本文作为后续，介绍我们目前的进展。

Linux 如何在 Secure Boot 下启动

UEFI Secure Boot 要求固件拒绝执行未被固件信任数据库中证书签名的启动代码。绝大多数支持 Secure Boot 的计算硬件都按微软的指南制造，其中包括把微软 UEFI 证书加载进固件信任数据库作为信任锚。

微软并不签署 GRUB 引导加载程序。相反，它签署 shim 二进制文件——这是第一阶段的引导加载程序，随后它加载发行版自己的引导加载程序（通常是 GRUB），后者再验证并加载内核。

从 shim 开始，信任链通过发行版自己的基础设施延伸。发行版管理自己的签名密钥，无需涉及微软就能发布 GRUB 和内核的更新。shim 被设计成很少变动，这样微软的签名角色就不会成为发行版发布或安全更新流程的一部分，除非 shim 自身需要被签名。

shim 的代码是开源的。社区运行一个 shim 审查流程，作为签名的主要关卡。一个发行版（如 Debian 或 Red Hat）在 GitHub 仓库中创建一个 issue，其中包含要签名的 shim.efi 二进制、构建日志，以及审查签名请求所需的额外信息。社区批准提交后，发行版才可以向微软请求签名；微软随后验证提交者与发行版的关系，运行自己的检查，并施加密码学签名。这种安排让社区无需运行签名基础设施、也无需让所有固件厂商都包含他们的证书，就能管理审查。我在这一过渡期间的部分工作，就是 bridging shim 审查社区与微软内部签署 shim 二进制的团队之间的沟通。

Microsoft Corporation UEFI CA 2011 证书已于 2026 年 6 月 26 日（或 6 月 27 日，取决于时区）过期。它正被 Microsoft UEFI CA 2023 证书取代，后者直到 2038 年才过期。这不会导致现有系统（例如那些在过期前已成功启动的系统）停止启动；然而，如果一个系统信任数据库中没有新证书、却通过系统升级或全新安装收到了一个用该证书签名的新的 shim，它就可能启动失败。

注意，作为各种信任路径信任锚的有多张 2011 证书。作为一个家族，这些证书从 6 月开始过期，最后过期的 Windows Secure Boot 证书则在 10 月。对 Linux 而言，重要的那张是 6 月 26 日过期的证书。

过期如何影响 Linux

固件在设计上通常会忽略证书过期，当它在评估签名是否有效时。UEFI 规范的镜像验证过程，其信任决策基于镜像的签名证书是否存在于授权数据库（db）中、且不存在于禁止数据库（dbx）中。它并不检查证书的有效期。用于签署 UEFI 二进制的中间证书其实已经过期而毫无后果。任何用 2011 证书签名的 shim 都将继续像从前一样工作，除非某个系统的特定固件实现有 bug。

2025 年 10 月，微软开始为每个 shim 提交返回两个已签名的二进制。一个用 2011 证书签名，另一个用 2023 证书签名。这种重叠随着证书过期而结束。未来的提交只能收到 2023 签名，留给发行版去决定：是在还能这么做时发布一个双签名的 shim，还是直接转向只用 2023 的 shim。后一种选择，只对那些固件已经信任 2023 证书的系统才安全。

不过，本文焦点在 shim 上；选项 ROM——也就是在预启动阶段被加载进显卡等外设并执行、作为 UEFI 二进制的那些东西——有着相似的暴露面。

被迫更新

UEFI 启动环境吸引了越来越多的安全审视。shim、GRUB 和其他启动链组件偶尔也会像其他软件一样被发现漏洞。如果在 shim 自身发现了一个需要代码改动和新二进制的漏洞，那新二进制只能用 2023 证书签名。

因此，依赖旧的 2011 签名 shim 的实际窗口，就是"现在"到"下一次强制 shim 重新构建的安全漏洞或发行版发布"之间的时间。Matthew Garrett 曾辩称，这次过渡没有某些描述暗示得那么紧迫：在许多情况下，系统可以启动较旧的介质、更新它的信任数据库，然后安装或启动较新的介质。这条路存在，但它并非对所有环境都足够。那些有严格正常运行时间要求、受限启动介质策略，或仅远程可访问的系统，可能就没有那个选项。

对大多数运行在受维护硬件、或受维护虚拟化层上的受维护发行版的用户而言，发行版会管理这次过渡。Linux 供应商固件服务（LVFS）和 fwupd 提供了一种交付固件更新的机制，包括 Secure Boot 信任数据库更新。

需要关注那些该路径不工作的系统：厂商已不再支持该型号、或已停止发布 LVFS 更新的硬件；以及在云或虚拟化提供商以 2023 证书登记之前就已配置的、且此后未收到固件更新的虚拟机。对这些系统，mokutil 可以报告 2023 证书是否已登记，而 pesign 或 sbverify 之类的工具可以显示安装的 shim 是由哪张证书签名的。升级信息见此帖。

LVFS 维护着它自己的兼容性数据，并能把更新定向到特定的硬件配置。微软发布了一组数据集，记录了 2023 证书更新已被确认能成功应用的设备配置，数据源自 Windows 机群遥测，可供 Linux 发行版和其他非 Windows 操作系统使用。

那些兼容性工作很重要，因为被迫的信任数据库更新并非在所有硬件上都安全。被迫更新曾让一些机器无法启动。在某些硬件上，恢复需要物理接触，而在某些情况下我们观察到系统被永久损坏。

发行版和计算厂商是当前状态和指导的最佳来源。shim 审查社区不是针对单个系统的支持渠道，而微软只能协助它自己的计算平台和操作系统。

总体而言，结果与其说是一次紧迫的、由截止日期驱动的宕机，不如说是一个预防性维护问题；系统管理员需要采取行动，以避免最终陷入系统可能停止启动的境地，但证书过期并不自动意味着系统在证书过期的瞬间就不再启动。

最有可能遇到麻烦的系统，不是最新或管理最细心的系统；而是那些在未来的某次 shim 更新变得必要时，被遗落在常规固件更新机制之外的系统。

存储

审视 MinIO 替代品：Ceph 与 Garage

作者：Tyler Langlois（特约撰稿人） 发布日期：2026年6月25日

MinIO 是一个流行的对象存储服务器，提供与 Amazon 简单存储服务（S3）API 的兼容性。2025 年 12 月，该项目的背后公司（同样名为 MinIO）宣布项目进入维护模式，不再接受新改动；它在 2026 年 2 月被完全归档。自那时起，MinIO 用户一直在寻找替代品，但众多选择可能令人眼花缭乱。虽然许多其他项目旨在填补这一空间，但它们的优势和关注点往往各异。其中两个替代品——Ceph 与 Garage——尤其引人注目，二者都提供扎实的 S3 兼容性。

MinIO 没有说明其退役项目的理由，但它有一个功能相似的专有项目，并推荐用它来替代现已停摆的项目。该公司持续发布漏洞信息，详述影响 MinIO 最后一版的安全问题，至少持续到今年 4 月，但已不再为开源项目提供更新。

大多数 MinIO 用户已经转向其他仍采用开源许可证的项目。Ruohang Feng 维护着一个名为 Silo 的 MinIO 分支，但他表示自己的计划仅限于 bug 和安全修复；新功能开发不在范围内。它最近的发布是在 2026 年 6 月 18 日。

核心需求

像 MinIO 这样的对象存储应用，提供了一种在自托管底层基础设施的同时提供 S3 能力的方式。对于无法依赖外部硬件与管理、处于隔离（air-gapped）环境的系统来说，那是必须的，但它对想避开专有服务的用户同样有益。然而，若没有支撑大多数商业服务的全球基础设施，在较小环境中高可用很难实现。幸运的是，许多项目（如 MinIO）从一开始就内置了容错设计。

易于部署、 modest 的硬件需求，以及 S3 API 支持，是基本要求。MinIO 成了一种流行的、为独立于云服务运行、却又依赖对象存储可用性的环境提供对象存储的方式，它常作为 MicroK8s 等 Kubernetes 发行版中的一个附加服务出现。

S3 API 因其跨多种用例的对象存储能力而被广泛采用。S3 通过 HTTP 实现基于文件系统的用例，用 GetObject 调用实现读取、用 PutObject 调用实现写入。

例如，S3 存储与许多 Kubernetes 环境一起使用，为容器镜像存储和容器卷的共享存储提供支持。它的用处当然不限于容器工作负载；对象存储常用于静态网站托管、缓存策略的后备存储等等。

MinIO 服务器的分布式特性帮助它在磁盘或主机故障时获得弹性。容错通过跨部署的基于复制的策略（在独立集群中保留多份数据副本）或基于纠删编码（erasure coding）的更高级能力来实现，后者通过集群内的奇偶校验更优地利用可用空间、以重建缺失数据。

MinIO 不合时宜的退役意味着用户需要新的替代品，但该选什么？韧性往往与 S3 兼容性同等重要，以防范数据丢失。然而，在满足这些要求的同时又 prohibitively 复杂或难以运维的解决方案，对寻找自托管选项的管理员而言会适得其反。因此，确保部署保持可管理很重要。

一个项目的 S3 API 实现质量是最值得评估的特性之一，但它不是唯一的考量。实现还必须通过公认的签名方法支持兼容的身份验证。当前标准是 AWS Signature Version 4；Version 2 已被弃用，但一些较老的客户端可能仍依赖它。Version 3 从未被广泛使用或支持。

虽然几乎所有对象存储解决方案都支持 GetObject 和 PutObject 等基本操作，但其他特性值得在个人需求语境下评估。例如，控制一个对象是否可在一段时间后删除的对象生命周期规则，可能并非每个 S3 替代品都支持。类似地，服务端对象加密（SSE）可能受支持，但它有若干不同模式，比如每次请求随附用户提供的密钥，对比服务器管理的密钥。

虽然 S3 API 会随着时间缓慢添加特性，但现有端点很少改变。客户端通常可以依赖既有 API 无限期工作、而新 API 极少出现。S3 API 用户指南描述了持续的变更。Ceph 或 Garage 等下游项目，多半是通过在端点被添加时拓宽与新能力的兼容性来工作的。多数与 S3 交互的库和命令行工具，只需极少的配置改动就能接受其他对象存储端点。

Ceph

Ceph 是一个稳定且长寿、治理健康项目。它现在由 Ceph 基金会管理，该基金会隶属 Linux 基金会。其源码托管在 GitHub 上，采用混合的开源许可证，最常见的是 LGPL 2.1。Ceph 也是历史悠久的 S3 兼容解决方案之一，拥有多年的开发历史和稳定的资金支持记录。

对 Ceph 的支持早在 2010 年就被合入了主线 Linux 内核。开发保持稳定，最新发布 20.2.1 于 2026 年 4 月宣布。与 MinIO 或 Garage 相比，Ceph 的显著特征是其面向海量规模、而非较小部署的能力。

S3 对象存储 API 只是 Ceph 存储能力的一个接口：构建于 librados 之上的 Ceph 对象网关提供 S3 兼容层，但 Ceph 的存储抽象也通过其他方式暴露：Ceph 有一个符合 POSIX 的文件系统（CephFS），它的块设备被 QEMU 和 libvirt 等其他开源项目用来为虚拟机提供后备存储。

分布式运行是 Ceph 架构的原生特性，以提供更好的容错性和可扩展性。Ceph 集群把操作分解为小的逻辑组件，可根据架构需求独立扩展。这让 Ceph 在付出一定运维复杂性的代价下获得了显著的灵活性：ceph-osd 守护进程管理实际存储操作，ceph-mgr 守护进程协调整体集群操作与管理，ceph-rgw（RADOS 网关）提供实际的 S3 API，如此等等。对有专用团队的大型部署而言，这些权衡或许说得通：如果某个子系统的关键路径出现瓶颈，它可以独立于其他系统扩展。

多个 Ceph 监视器组成一个 Paxos 法定人数（quorum），或称多数共识，以确保一致性并协调集群活动。ceph-mon 工具借助对初始引导成员的了解（这类算法在初始设置期间很典型）来帮助管理配置过程。像 ceph-deploy 这样的工具可以帮助简化初始配置过程。

与其他依赖基于法定人数协调的集群服务一样，Ceph 监视器应当以奇数个投票成员运行，以避免脑裂（split-brain）情况。三个 ceph-mon 实例允许丢失一个成员而不降级运行，五个则允许两个成员离线，依此类推。任意时刻只有一个 ceph-mgr 处于活跃状态，一般建议每个 ceph-mon 守护进程运行一个实例以承接潜在的故障转移职责。其他守护进程随硬件一起扩展：ceph-osd 通常对应一个硬件存储设备。虽然 Ceph 可以作为单节点运行，但它的默认运行模式更适合多台主机。

系统硬件需求是微妙的：因为 Ceph 支持潜在大规模的集群，所需资源是整体期望有效存储量的函数。像 ceph-mds（元数据服务器）这样的组件只需 1GB 内存就能工作，但承担 ceph-mon 和 ceph-mgr 角色的节点可能以 64GB 表现良好、而当 ceph-osd（存储守护进程）数量开始超过数百个时需求会上升到 128G。虽然其中一些数字可能令人望而生畏，但 Ceph 正是被设计成能在这些规模上出色运转的。

Ceph 的 RADOS 网关提供了成熟的 S3 兼容 API：它支持桶生命周期、加密，以及 S3 生态原生的其他概念。依赖最像 S3 的 API 的管理员，很可能会发现它的实现很有吸引力，尤其是当它与访问控制列表（用于微调权限）和桶通知（用于对桶对象活动做出反应）等运维工具结合时。该项目维护着一个概述 S3 API 兼容性的列表，包括同时支持 version 4 和 version 2 签名。

Garage

在长寿光谱的另一端，Garage 是一个相对年轻的 AGPL 许可项目；它由非营利托管公司 Deuxfleurs 于 2020 年启动。项目托管在独立的 Forgejo 基础设施上。

Garage 用 Rust 编写，它放弃庞大的特性目录，而专注于一个明确的目标：分布式、容错的对象存储（没有任何传统文件系统或块设备驱动）。撰写时的最新稳定版本是 2026 年 4 月的 2.3.0。虽然相对较新，Garage 的定位是居于 Ceph 那不算简单的部署架构，与缺乏分布式容错、却更简单的项目之间的中间地带。

Garage 运行同构服务，并依赖无冲突可复制数据类型（CRDT）之类的概念，在网络分区等事件发生时协调一致性漂移。就像把共识委托给 Paxos 这样的外部协议，所涉及的原语往往是健全的，但实现可能缺乏来自多年实际使用的运维记录。对于只有一个管理员的小型环境或家庭实验室，既提供可靠性保证、又极其简单的配置很有吸引力。

组建一个 Garage 集群比配置 Ceph 监视器要省事：运行节点会用 garage node id 命令给出自己的标识符和地址，随后对等节点用 garage node connect $node_id（给定目标节点 ID）协商连接。这让 Garage 部署具有一定弹性，不过运维者应当记住，某些操作仍会咨询内部法定人数（其设计文档提及了 Amazon 的 Dynamo 论文）。因此，保持多数 Garage 节点在线对整体集群稳定性仍然重要。

Garage 把所有功能封装在单一守护进程中，使它很适合较小或资源受限的环境。部署拓扑通常遵循：每台主机一个能够管理多个物理磁盘、向整体集群呈现的守护进程。不过，对于磁盘维护频繁的大型网络，单一的 ceph-osd 守护进程可能是更合理的选择，它不需要打断同时管理集群成员资格和存储设备的单一进程。系统需求极低，支持的目标是最近 10 年内发布的 CPU 和最少 1GB 内存。

Garage 的 S3 兼容性页面直截了当，并维护着与其他 S3 API 实现的对比表。虽然项目演进很快，Garage 的 API 支持仍是整体 S3 API 中较小的一个子集。特别是，一些缺失的能力（如对象版本控制，例如用 PutBucketVersioning 隐式创建对象备份副本）可能是基线需求，而且只支持 version 4 身份验证签名。Ceph 提供更广的 S3 API 支持，但运维成本可能值得用 Garage 能提供的、要求更可控的简单性来交换。

真实世界经验

对 Ceph 和 Garage 而言，总可用存储都是一个相对简单的计算。在不做任何复制配置的情况下，任一方案都把所有汇集设备呈现为一个逻辑视图：例如，三块 1TB 磁盘呈现为 3TB 可用空间。以因子 2 复制会为托管数据保留一份副本，把可用空间减半到 1.5TB。Ceph 对卷做纠删编码的能力，通过奇偶校验计算提供更好的容量，在这个假想场景中会提供大约 2TB 可用空间。我自己的 Garage 基础设施横跨六台主机、总共十个物理磁盘。

Garage 至今未曾灾难性失败，而我曾因不可调和的集群问题被迫至少重建过一次我的 MinIO 集群。总体而言，我会把 Garage 描述为比 MinIO 更易运维、更易使用。不过，用例各有不同；而像 Ceph 这样庞大却强悍的解决方案，几乎肯定更适合那些有更严格需求、需要更接近的 S3 API 对等性、或更强稳定性保证的环境。

尽管 MinIO 退出了开源版图，替代品们提供了成熟实现，可以在不牺牲许多 S3 原生能力的前提下提供重要特性。Ceph 和 Garage 都是适合内部自托管环境的能干的替代者。

开发实践，尤其是涉及大语言模型（LLM）使用的部分，也已成为考量项目维护时的一个突出问题。Ceph 项目仍在就其 LLM 政策进行辩论，但似乎正收敛到与 Linux 内核 AI 贡献政策类似的东西。Garage 最近合入了一项关于 LLM 及其他编码助手撰写贡献的政策；它允许把 LLM 用于"一些繁琐的代码生成任务"，但大体禁止把它们用于文档、撰写 bug 报告，或大多数要提交给项目的代码。虽然 Garage 仓库里没有任何由 LLM 撰写或共同撰写的提交记录，Ceph 的提交历史中却已越来越多地出现由 Claude、ChatGPT 和 Gemini 共同撰写的代码。

据我个人的经验，从 MinIO 迁移到 Garage 的过程是积极的，没有任何负面意外。集群组建容易，从硬件故障中恢复可靠，而且它的 API 与我所有的用例都兼容。Garage 很适合我的家庭实验室，其他小规模运维者大概也能从多台服务器提供的故障转移中受益，而无需承担编排相对更复杂部署的心理负担。无论对于 MinIO 还是 Garage，我的集群都经历过滚动更新和偶尔的坏盘，同时保持运行。

联邦宇宙（fediverse）和去中心化社交网络总体而言是对中心化的一点反制。问题是，在做去中心化技术时，我一向相信它很重要，因为我们有来自政府和企业的中心化、监控等真正风险，它们可能合谋。但我没预料到的是：随着事情变得更中心化，为互联网作为公共利益而战的意志也会消散。

当我和那些还没怎么想过年龄验证及类似法案及其后果的家人朋友交谈时，他们会说"好吧，总得有人来制约像 Meta 这样的企业"。对此我会说："可互联网上所有更小、非企业的部分呢？"对此许多人感到惊讶，因为他们早已忘了那些东西的存在。

当互联网和计算对大多数人变成五家企业时，他们就开始把它当作"制约五家企业"的事情。

—— Christine Lemmer-Webber

开发

Git 2.55.0 发布

Git 维护者 Junio Hamano 宣布 Git 2.55.0，包含来自 100 人的非合并提交，其中 33 位是项目的首次贡献者。LWN 近期报道了 2.55 中一些值得注意的改动，包括实验性 git history 命令的新特性，以及为 Linux 系统新增的 Git fsmonitor 守护进程等。

AI 时代的开源维护（Kubernetes 博客）

Kubernetes 项目发布博客解释其 AI 政策。核心问题是：AI 让生成代码变快，但对维护代码库的改进甚少。项目要求在使用 AI 工具辅助创建贡献时予以披露，但禁止把 AI 列为共同作者、或包含 "assisted-by" / "co-developed" 尾注来把功劳归于 LLM 工具。

Podman 6.0 发布

容器管理工具 Podman 6.0.0 已发布。值得注意的新特性包括：为容器设置多个静态 IP 地址、改进网络隔离以更兼容 Docker、改变 Quadlet 命令的工作方式、许多现有 podman 命令的新选项，以及对 Podman 配置文件处理的重写。包含大量破坏性变更，详见发布说明。
