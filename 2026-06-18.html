2026年 Fedora 现状

作者：Joe Brockmeier 2026年6月16日

本文内容来自 Fedora 项目领导者 Jef Spaleta 在 2026 年 Flock 大会（布拉格举办）的主题演讲，涵盖 Fedora 使用现状、贡献者增长挑战、以及项目未来规划。

📈 增长亮点

  • 整体增长：Fedora 使用量同比增长 9%，过去一周有接近 100 万系统上报统计
  • KDE Plasma 版：刚升级为正式版本，同比增长超 120%，过去一周有 15.8 万活跃系统
  • Workstation (GNOME) 版：同比增长 18.8%，过去一周有 29.7 万活跃系统
  • 衍生发行版：基于 Fedora 的衍生版（如 Bazzite、Asahi Remix）同比增长超 210%，过去一周有 9.7 万活跃系统
⚠️ 异常点：Fedora Cloud 版活跃系统同比下降近 65%，统计数据显示异常波动，原因尚未明确。

🚨 核心挑战

活跃打包者数量持续下降：

  • 2020 年初：超 420 名打包者
  • 2023 年底：不足 340 名
  • 2026 年初：350 名 → 目前仅余 325 名

Spaleta 表示尚未明确下降原因，后续计划通过以下方式解决问题：

  • 升级贡献者体验（迁移到基于 Forgejo 的 Fedora Forge 平台）
  • 优化治理流程
  • 拓展捐赠渠道
  • 推出创新沙盒机制（参考 CNCF 沙盒项目，降低创新试错成本）
📌 争议事件：容器化滚动发行版 Fedora Hummingbird 未经过公开讨论流程，由理事会秘密投票批准使用 Fedora 商标，引发社区争议。

7.2 内核自动创建 mTHP 功能

作者：Jonathan Corbet 2026年6月11日

Nico Pache 贡献的新特性将 khugepaged 内核线程的页面合并能力从仅支持 PMD 级大页扩展到支持多尺寸透明大页（mTHP），提升 mTHP 的使用透明度。

背景

传统大页由硬件页表层级决定，尺寸固定（通常 2MB），存在内存碎片化、内部碎片等问题；mTHP 支持软件定义的 2 的幂次尺寸，更灵活，能更好匹配处理器 TLB 合并能力、降低内存管理开销。

实现逻辑

khugepaged 以 2MB 为单位扫描进程地址空间：

  1. 先尝试合并为 PMD 级大页
  2. 失败则逐步尝试更小的 mTHP 尺寸
  3. 直到最小支持尺寸，避免内存浪费

问题解决

针对合并过程中可能出现的"尺寸蠕变"问题（多次扫描后小 mTHP 被逐步合并为大页，浪费内存），调整了 max_ptes_none 参数的限制规则：

  • 仅允许 0 或 511 两个取值用于非 PMD 尺寸的 mTHP 合并
  • 避免无效合并
该补丁已迭代 19 个版本，进入 linux-next 分支，预计合入 7.2 内核合并窗口。

overlayfs 更新

作者：Jake Edge 2026年6月12日

来自 2026 年 Linux 存储、文件系统、内存管理与 BPF 峰会的分享，由 overlayfs 联合文件系统维护者 Amir Goldstein 介绍最新进展。

新特性

  • composefs 用例支持：新增仅数据底层、支持 fs-verity 完整性校验，适配内容寻址存储场景
  • 新挂载 API：切换到新挂载 API，解除底层数量和路径长度限制
  • 安全性提升：支持挂载凭证与访问凭证分离

嵌套 overlayfs 支持

  • 目前仅支持底层为 overlayfs
  • 后续计划支持上层也为 overlayfs，解决容器内运行 Docker 时的存储驱动性能问题
  • 考虑通过扁平化底层信息的方式避免嵌套导致的内核栈溢出问题

社区需求

  • systemd 希望获得 overlayfs 层的自省 API，方便追踪文件来源、管理 sysext 分层更新
  • 用户希望支持删除白名单后恢复底层文件的能力

buffer heads 清理工作

作者：Jake Edge 2026年6月17日

来自 2026 年 LSFMM+BPF 峰会的分享，内核开发者 Jan Kara 介绍 buffer heads（块层元数据跟踪机制，历史悠久）的清理进展。

已完成工作

  • 已将元数据 buffer heads 跟踪逻辑从通用 inode 结构剥离到文件系统私有部分
  • 未使用该功能的文件系统 inode 可减小 40 字节
  • 该改动已合入 7.1 内核

进行中工作

  • 清理 Amiga Fast File System (AFFS) 的 buffer heads 跟踪代码（该文件系统从未实际使用跟踪信息进行 fsync 操作）
  • 修复 buffer heads 跟踪的竞态问题（可能导致 fsync 后元数据未写入磁盘，崩溃后数据丢失）
  • 除 ext4 外其他使用该机制的文件系统均受影响

未来计划

  • 计划未来一年内重构 buffer heads 的锁机制，使用 RCU 替代现有的私有锁，降低复杂度
  • ext4 data=journal 模式下的脏 buffer heads 脱离映射问题
  • 计划修改日志写入路径，使用标准回写机制处理，避免数据处于"中间状态"
