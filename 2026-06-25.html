自由线程 Python：过去、现在和未来

作者：Jake Edge 发布日期：2026年6月22日

核心背景

过去5年 Python 最重大的变更是自由线程（free-threaded）版本的推出，该版本移除了全局解释器锁（GIL），支持多线程并行执行。CPython 核心开发者、Python 指导委员会成员 Thomas Wouters 在 PyCon 2026 上分享了该特性的开发动机、历史、当前状态与未来规划。

动机：为什么要移除 GIL

线程的核心价值：多线程共享进程地址空间，可提升性能（弥补内存访问延迟）、支持阻塞 API 调用、适配需要线程的第三方库（如数据库驱动）。

GIL 的历史局限性：GIL 是 CPython 早期支持线程的方案，用于保护 Python 对象引用计数与内部状态，但会导致多线程无法并行，随着多核 CPU 普及，GIL 已成为性能瓶颈。

现有替代方案的不足：

  • 多进程：内存开销大、数据拷贝成本高；
  • 子解释器：跨解释器数据拷贝困难，第三方库兼容性差；
  • asyncio：适合网络 I/O，但无法覆盖所有高性能场景。

Python 的固有挑战：所有对象都是共享的可变数据，引用计数的线程安全修改需要大量锁，会导致单线程性能下降 50%，无法接受。

移除 GIL 的历史尝试

  1. 1996年 Greg Stein 为 Python 1.4 提交的补丁：用细粒度锁替换引用计数，单线程性能损失过大，未成功。
  2. 2013年 PyParallel 项目：采用非操作系统线程模型，无法兼容现有 OS 线程生态，不是通用方案。
  3. 2015年 Gilectomy 项目（Larry Hastings 主导）：尝试了多种引用计数优化方案，最终因性能与扩展性问题放弃。
  4. 近年 Meta 的 Sam Gross 开发的 No-GIL CPython 3.9 分支：成为当前自由线程 Python 的基础，对应的 PEP 703（让 GIL 在 CPython 中可选）描述了移除 GIL 所需的核心技术。

PEP 703 的核心技术方案

  • 偏向引用计数：对象有"所属线程"，所属线程修改引用计数走快速路径，其他线程走基于原子操作的慢速路径。
  • 延迟引用计数：部分引用计数操作延迟到垃圾回收阶段统一处理，减少锁竞争，仅会导致对象回收时机略有延后。
  • 乐观列表/字典访问：访问容器成员时先增量引用计数，再校验对象是否仍存在于容器中，通过可控的安全机制保证正确性。
  • 基于静止状态的内存回收：列表、字典释放后内存不会立即复用，直到所有线程都进入静止状态，避免其他线程访问已释放对象的引用计数。
  • 细粒度锁：为必要的操作添加细粒度锁，替换原有 GIL。
  • 新的垃圾回收器：适配自由线程场景，替换原有复杂的垃圾回收逻辑。
  • 内存分配器切换为 mimalloc：线程安全，且支持静止状态回收所需的钩子。
  • 无锁容器：实现无锁的列表、字典等核心类型，保证所属线程访问速度，其他线程访问也保持较高性能。
  • 全停顿（Stop-the-world）：修改对象类型等需要全局一致性的操作时，暂停所有线程完成修改后再恢复执行。
  • 每对象锁与临界区：实现无死锁的临界区，复用原有 GIL 的获取/释放接口，保证"停止世界"操作可以正常完成。

当前状态

  • Python 3.13（2024年10月发布）：实验性支持自由线程，单线程性能下降 20-40%，存在线程安全问题与扩展性瓶颈，验证了方案可行性。
  • Python 3.14（2025年10月发布）：自由线程版本安全性与性能大幅提升，单线程性能下降 0-10%（Arm 硬件上可实现 0 下降，Linux GCC 环境下通常 5% 左右，最高 10%），指导委员会通过 PEP 779，移除自由线程的"实验性"标签。
  • Python 3.15（2026年10月即将发布）：
    • 统一 GIL 版本与自由线程版本的 ABI，扩展开发者只需编译一次，即可同时支持两种 Python 版本。
    • 大量扩展性优化。
    • 提供自由线程移植指南，帮助扩展开发者适配自由线程。
  • 生态现状：PyPI 上排名前 360 的二进制 wheel 中，超过 50% 已经支持自由线程。

未来规划

持续优化自由线程版本的性能与扩展性，推动更多第三方包支持自由线程。Wouters 预测：自由线程版本将在 Python 3.16（2027年10月）之后、3.20（2031年10月）之前成为 Python 默认版本，GIL 版本将在 2030 年代完全移除。

扩展适配建议：扩展开发者需要：

  1. 保护 C 全局变量（GIL 不再提供保护）；
  2. 使用临界区保护 Python 对象的可变数据；
  3. 声明扩展支持自由线程；
  4. 添加多线程测试，暴露潜在的线程安全问题。

普通 Python 代码：通常无需修改，但如果原有代码存在线程安全 bug，自由线程版本会更易触发问题，建议添加多线程测试。

AURpocalypse now：近期 AUR 攻击分析

作者：Joe Brockmeier 发布日期：2026年6月19日

事件概述

Arch Linux 的用户软件仓库（AUR）近期遭遇大规模持续攻击：攻击者注册大量新账号，接管无人维护的软件包（orphaned packages），推送包含恶意代码的更新，试图在用户设备上安装恶意软件。Arch 已暂时关闭 AUR 新用户注册，长期解决方案仍在讨论中。

AUR 的固有脆弱性

  • AUR 定位：存放未被纳入 Arch 官方仓库的用户贡献软件包，提供 PKGBUILD 文件供用户自行编译，无官方审核流程，目前有超过 10.7 万个软件包，其中近 1.4 万个为无人维护状态，注册用户超过 14.1 万。
  • 接管规则：任何注册用户都可以一键"收养"无人维护的软件包，无需任何审核，攻击者可轻易批量接管大量包。
  • 与同类服务的差异：Fedora Copr、openSUSE OBS、Ubuntu PPA 等类似服务均采用用户独立命名空间，其他用户无法接管他人的仓库，而 AUR 所有包都在统一命名空间下，仅依赖用户信任维护者信息，极易被滥用。

攻击详情

  • 历史攻击：2018年、2025年均有小规模 AUR 攻击事件，影响范围有限。
  • 本次攻击时间线：
    1. 2026年5月27日：首次发现攻击，plex-media-player 包被篡改，植入恶意 npm 包 crypto-javascript。
    2. 2026年6月11日：攻击升级，攻击者批量接管数百个无人维护包，例如 gnome-randr-rust 包被添加恶意依赖 atomic-lockfile，该恶意代码包含 eBPF 程序，可窃取 GitHub 凭据、SSH 信息、浏览器 Cookie、聊天软件数据等。
    3. 2026年6月13日：攻击者切换为使用 Bun 包管理器安装恶意依赖，并混淆安装命令规避扫描脚本。
  • 影响范围：超过 1500 个软件包被篡改，受影响用户数量暂未明确。

当前响应与后续讨论

临时措施：2026年6月11日起关闭 AUR 新用户注册，尝试添加 Anubis 防护对抗批量注册未生效，目前注册仍处于关闭状态。

社区讨论的解决方案：

  • LLM 辅助扫描：使用小型本地 AI 模型扫描 PKGBUILD 文件，检测恶意代码；
  • 无人维护包锁定：禁止随意收养无人维护包，需要提交申请说明更新理由，但该方案可能阻碍正常维护者接手；
  • AUR 工具侧增强：比如在包被收养、近期修改时向用户发出醒目警告，目前 yay 已经在新版本中显示 PKGBUILD 的最后修改时间，提醒用户审查。

核心矛盾：AUR 的开放协作模式与安全防护存在天然冲突，完全审核会破坏社区活力，不审核则无法应对规模化攻击。

Fedora：2FA，还是不 2FA，这是个问题

作者：Joe Brockmeier 发布日期：2026年6月24日

事件背景

账户泄露是开源软件供应链攻击最常见的入口，2026年5月 Fedora 发生疑似账户泄露事件，一个未监督的 AI 代理在 Fedora 项目中提交了无意义贡献，虽未造成危害，但暴露了账户安全问题。Fedora 因此重新讨论强制要求贡献者启用双因素认证（2FA）的方案。

讨论过程

  • 历史背景：2024年 XZ 后门事件后，Fedora 曾提出强制 2FA 的方案，但被 FESCo（Fedora 工程指导委员会）弱化，仅要求 provenpackager 组（可修改所有 Fedora 官方包的特权贡献者组）成员"应该"启用 2FA，未强制要求。
  • 反对意见：部分开发者认为 Fedora 的 2FA 用户体验不完善，比如 GNOME 在线账户功能不支持 Kerberos 2FA，而 Fedora 部分基础设施使用 Kerberos 认证，会影响工作流程。
  • 支持意见：认为账户泄露的风险远高于体验问题，强制 2FA 可以推动相关用户体验问题的修复，至少应该先要求 provenpackager 组启用 2FA，该组账户泄露会影响所有 Fedora 用户。

最终决定

2026年6月23日 FESCo 会议投票通过决议：

  • 要求 provenpackager 组的 114 名成员在 3 个月内启用 2FA，逾期未启用的成员将被从组中移除。
  • 当前 Fedora 暂无完善的 2FA 恢复机制，用户丢失 2FA 设备可通过邮件联系管理员团队恢复账户，后续会优化恢复流程。
  • 未来会逐步推进所有 Fedora 打包者强制启用 2FA。

7.2 合并窗口前半段

作者：Jonathan Corbet 发布日期：2026年6月18日

背景

Linux 7.1 内核于 2026年6月14日发布，7.2 合并窗口开启，截至本文发布时已有超过 7000 个非合并变更集被合入主线，大部分核心子系统的变更已确定。

主要变更汇总

架构相关

  • 移除 i486 的浮点单元模拟代码，删除超过 1.3 万行代码。
  • AMD Geode 处理器（用于 OLPC XO-1 电脑）被标记为孤儿状态，不再有官方维护。
  • Intel TDX（可信域扩展）支持增强：新增管理模块的设备类型，支持在运行系统中替换 TDX 模块（用于安装安全更新）。
  • s390 架构新增 Rust 支持。
  • arm64 内核尝试将数据区域从全内存线性映射中移除以提升安全性，因 KVM 回归暂时回退。
  • 移除 PowerPC 架构的硬件优化 MD5 实现（MD5 安全性已不足）。

BPF 相关

  • 附加到跟踪点的 BPF 程序现在允许缺页，可可靠访问用户空间内存。
  • bpf() 系统调用新增通用属性支持，首先用于部分 BPF 子命令的日志控制。
  • 移除 BPF 函数与 kfunc 的参数数量限制（原限制为 5 个），允许额外参数放在栈上。
  • 内核代码现在可以安全访问 BPF Arena，无需担心缺页问题。
  • 新增可调整大小的 BPF 哈希表类型，不再受创建时固定桶数限制。
  • 支持快速将同一个 BPF 程序附加到多个跟踪点。

文件系统与块 I/O

  • 文件系统现在可通过 file_getattr() 系统调用提供大小写敏感性信息，新增两个标志：FS_XFLAG_CASEFOLD 表示文件名查找不区分大小写，FS_XFLAG_CASENONPRESERVING 表示不保留新文件名的大小写。
  • openat2() 系统调用新增两个标志：
    • O_EMPTYPATH：允许重新打开 O_PATH 文件描述符对应的文件；
    • OPENAT2_REGULAR：如果目标不是普通文件则打开失败，返回新错误码 EFTYPE，用于加固系统防止被诱导打开特殊文件。
  • XFS 的 zoned 存储设备支持在 7.2 中不再是实验性特性。
  • Btrfs 默认使用大 folio，新增最大 2MB 的巨型 folio 实验性支持。
  • 新增 dm-inlinecrypt 设备映射目标，支持带内联加密能力的块设备。

网络相关

  • TCP 认证选项的实现切换到新的 libcrypto 库，简化代码、提升效率，移除未使用的算法支持。
  • 多路径 TCP 连接支持的最大子流数量从 8 提升到 64。
  • 继续推进减少 rtnl_lock（网络大锁）的使用，提升网络栈扩展性。
  • 移除过时网络支持：ISA/PCMCIA ARCnet 接口、PCMCIA 蓝牙适配器、Chelsea 内联 TLS 加速器、部分 ATM 支持、AppleTalk 协议。

安全相关

  • slab 分配器支持使用 Clang 分配令牌，隔离不同类型分配对象的内存布局，提升缓冲区溢出利用难度。
  • 不常用的 AF_ALG 加密卸载机制被弃用，计划未来移除，同时移除硬件加速器支持，仅保留软件实现，移除大量过时的随机数生成器驱动。

内核内部变更

  • 新增 bh_submit() 函数提交缓冲区头 I/O，移除 b_end_io 回调指针，提升性能与安全性。
  • iomap 层新增 fs-verity 支持。
  • 新增文件系统开发者文档：Documentation/filesystems/adding-new-filesystems.rst。
  • 构建内核的最低 LLVM 版本要求提升到 17.0.1。
  • 新增 kconfig-sym-check 构建目标，检查 Kconfig 文件中引用未定义的符号。
  • 内核引入 Rust 的 zerocopy 库，提供低成本内存操作原语，隔离不安全代码。
  • Rust 新增高阶生命周期类型，更好地封装驱动生命周期与绑定设备的关系。

后续安排

7.2 合并窗口将持续到 2026年6月28日，目前 linux-next 中还有约 6000 个变更集等待合入主线，合并窗口关闭后 LWN 会发布完整变更汇总。

BPF Arena 辅助库

作者：Daroc Alden 发布日期：2026年6月24日 关联会议：2026年 LSFMM+BPF 峰会

背景

BPF Arena 是可供 BPF 程序自由构建自定义数据的内存区域（可与用户空间共享），无需验证器的边界检查，但当前缺乏通用的内存分配器与数据结构实现。Emil Tsalapatis 在峰会上介绍了他开发的 libarena 库，为 BPF Arena 提供通用工具。

libarena 的定位与规划

  • 定位：作为 BPF 程序的标准库，提供通用的内存分配器与常用数据结构，代码放在内核树中，编译后链接到 BPF 程序，可选使用，不会增加小型 BPF 程序的负担。
  • 当前状态（7.2 内核）：实验性可用，包含伙伴分配器与 ASAN（地址 sanitizer）钩子，暂无数据结构实现，分配器性能有待优化。
  • 未来规划：
    1. 分配器可能切换为 mimalloc 设计，提升多线程与短生命周期分配的性能；
    2. 提供红黑树、B树、位图、Chase-Lev 队列等常用数据结构；
    3. 完善 ASAN 支持，帮助检测 Arena 中的内存安全问题。
  • 兼容方案：libarena 会保证接口稳定，用户可针对自己需要支持的最低内核版本编译 libarena，无需 CO-RE 重定位。
BPF

挂起和恢复 BPF 程序

作者：Daroc Alden 发布日期：2026年6月19日 关联会议：2026年 LSFMM+BPF 峰会

背景

当前 BPF 程序必须在启动的同一上下文中运行至完成，无法挂起与恢复，导致跨时间、跨上下文的逻辑任务（比如收集用户态与内核态联合栈轨迹）需要拆分成多个独立函数，开发难度大。Kumar Kartikeya Dwivedi 正在为 BPF 引入协程支持，允许 BPF 程序挂起、交还控制权给内核、后续再恢复执行。

设计方案

采用无栈协程设计（类似 Rust 异步函数、C++ 协程）：编译器将协程代码重写为可通过 resume() 恢复、destroy() 销毁的结构，挂起时的中间变量保存在协程结构体中，验证器无需大改即可支持。

验证器需要新增的校验：确保 resume() 与 destroy() 函数指针未被篡改、协程状态索引合法、挂起时可安全调用 destroy()、挂起时未持有锁或引用计数资源。

当前进展与规划

  • 已有原型实现，后续需要完善 BTF 调试信息支持、添加 Rust 支持。
  • 更长远的实验性规划：允许挂起的计算在用户态与内核态之间切换，模糊用户态与内核态的边界，但该方案离落地尚远。

JIT 编译的 BPF 代码的 KASAN 支持

作者：Daroc Alden 发布日期：2026年6月23日 关联会议：2026年 LSFMM+BPF 峰会

背景

KASAN（内核地址 sanitizer）是内核内存错误检测工具，可检测释放后使用、越界访问等问题，但此前 JIT 编译的 BPF 代码未添加 KASAN 插桩，无法检测 BPF JIT 编译器的内存 bug。Alexis Lothoré 正在为 JIT 编译的 BPF 代码添加 KASAN 支持。

实现方案

在 BPF JIT 编译阶段，为 LDX、STX 等内存访问指令插入 KASAN 的 __asan*() 检查函数调用，检测内存访问合法性。

当前进展：x86 架构的原型补丁已实现，仅支持基础加载/存储指令，不包含栈访问与原子指令，单条指令会扩展为 12 条指令，性能开销较大，后续可通过内联 __asan*() 函数优化。

后续规划

  • 扩展支持更多 BPF 指令（如原子指令）；
  • 优化性能，降低开销；
  • 完善错误报告，确保 KASAN 报告可关联到对应的 BPF 程序与代码行。

单跳块复制：RMR 和 BRMR

作者：Haris Iqbal 发布日期：2026年6月18日 关联会议：2026年 LSFMM+BPF 峰会

背景与目标

为云厂商提供低开销的持久虚拟块设备方案，解决现有方案的缺陷：

  • DRBD：写入需要经过主节点，是两跳路径，性能差；
  • MD-RAID 1 + 网络块设备：恢复时需要从计算节点读取健康副本数据，占用计算节点资源。

目标：计算节点仅参与正常 I/O，恢复流程完全在计算节点外的存储节点之间完成，所有 I/O 仅经过一跳网络传输。

方案架构

方案基于 RDMA 传输库 RTRS 实现，包含两个内核模块：

  1. RMR（Reliable Multicast over RTRS）：传输层，提供多播消息发送能力，不知道上层应用逻辑。
  2. BRMR（Block device over RMR）：块设备适配层，通过内核 blk-mq 层暴露 /dev/brmrX 块设备，管理磁盘元数据。

核心设计

  • 复制池：一个 BRMR 设备对应一个 RMR 池，包含一个计算节点客户端与多个存储节点，每个存储节点对应一个 RMR 会话，无主节点，正常状态下读请求轮询发往所有健康节点，写请求多播到所有健康节点，所有节点都确认后才向应用返回成功。
  • 脏映射（dirty map）：每个存储节点保存所有成员的脏映射，记录故障节点错过的 I/O，采用两级页表设计，每个块对应一个字节的元数据，最多可跟踪 5.12 亿个块，单池最大支持 512TB 容量，元数据最多占用 512MB 内存。
  • 故障恢复：节点故障恢复后，从其他健康节点获取最新脏映射，仅同步脏数据，同步时不影响正常 I/O，无需计算节点参与。
  • 计算节点故障处理：每个存储节点记录最近的 I/O 操作信息，计算节点故障恢复后，可比对各节点的 I/O 记录，解决部分节点写入成功、部分失败的数据不一致问题。

当前状态

方案仍在开发阶段，需要社区反馈后才会向上游提交补丁。
